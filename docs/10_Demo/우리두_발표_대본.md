## 2) 차별화된 신뢰 포지셔닝

### 핵심 메시지
우리두의 차별점은 "선충전 락 + 투표 승인 + 기술적 강제"를 동시에 적용한 신뢰 구조다.

### 말하기 대본
이 페이지는 우리두의 포지셔닝을 한 장으로 보여줍니다.  
아임인과 카카오 모임통장, 일반 커뮤니티가 각각 일부 문제만 해결한다면, 우리두는 운영 리스크 통제, 통합 UX, 기술적 신뢰 강제를 동시에 제공합니다.  
핵심은 리더 개인 재량을 줄이고, 지출 승인과 기록 흐름을 시스템 규칙으로 강제하는 것입니다.  
즉, "좋은 사람이 운영하면 안전한 서비스"가 아니라, "누가 운영해도 안전한 구조"를 지향합니다.

### 기획↔구현 병기
- 기획 문서의 Trust Triangle(Deposit Lock, Transparency, Consensus Pay) 서사를 유지하되, 발표 수치/동작은 구현 산출물 기준으로 설명합니다.
- 예를 들어 지출 통제는 기획의 정책 설명에 더해, 실제 구현된 `VOTE_007` 제약과 에러 처리 흐름까지 연결해서 설명합니다.

우리두의 기획 의도는 리더 독단 리스크를 구조적으로 줄이는 신뢰 삼각형 설계다.

### 말하기 대본

넷째, 시연 계획과 팀 회고입니다.

보증금 락으로 이탈 리스크를 관리하고, 장부 투명성으로 신뢰를 공개하며, 합의 기반 결제로 지출 권한을 분산합니다.

## 5) 핵심 기능 도출

### 핵심 메시지
P0/P1/P2로 기능을 단계화하고, 상태 모델을 정규화해 시연 안정성을 확보했다.

### 말하기 대본
핵심 기능은 P0, P1, P2로 나눠 도출했습니다.  
P0는 충전, 챌린지 생성/가입, 지출 투표, 장부 조회처럼 시연의 뼈대가 되는 기능입니다.  
P1은 모임 흐름과 권한 박탈 등 안정성 강화를 위한 필수 확장이고, P2는 강퇴/종료 투표 같은 운영 고도화 영역입니다.  
또한 상태 모델은 투표 상태를 PENDING 중심으로 정규화해, 레거시 OPEN/IN_PROGRESS 표현 차이로 인한 혼선을 줄였습니다.  
BRIX는 월 배치와 수동 트리거를 모두 열어 운영 안전장치를 두었습니다.

### 기획↔구현 병기
- 기획 문서에는 `IN_PROGRESS/CONFIRMED` 계열 표현이 남아 있지만, 구현 산출물은 `PENDING` 정규화 및 실제 서비스 상태머신 기준으로 설명합니다.
- 배치 시점도 정책 문서의 00:00 표기와 달리, 런타임 설정과 운영 문서는 매월 1일 03:00 KST를 사용합니다.

## 6) 시스템 아키텍처 및 보안

### 핵심 메시지
Spring+Oracle이 트랜잭션을 책임지고, Django는 분석 계산을 분리해 보안 경계를 이중화했다.

### 말하기 대본
아키텍처는 Client-Spring-Oracle을 메인 경로로 두고, Django는 분석 엔진으로 분리했습니다.  
즉, 비즈니스 트랜잭션은 Spring과 Oracle에서 처리하고, BRIX/그래프 계산 같은 분석성 연산만 Django와 연동합니다.  
내부 연동은 `X-Internal-Api-Key`와 API 키 헤더를 사용해 프레임워크 간 경계를 분명히 했고, 운영 시나리오에서 키 불일치/장애 대응 절차를 별도로 관리합니다.  
권한 제어는 보안 필터 레벨과 서비스 로직 레벨을 함께 보며, 현재 구현은 서비스/컨트롤러 레벨 검증 비중이 큰 구조입니다.

### 기획↔구현 병기
- 기획 아키텍처 문서는 확장 관점(검색/추천 포함)을 크게 제시하지만, 시연 발표는 실제 구현된 내부 엔드포인트와 운영 런북 중심으로 설명합니다.
- 특히 보안은 문서상 이상형보다 현재 런타임 제약(`permitAll` + 서비스단 권한검증)을 투명하게 병기해 설명합니다.

## 9) 시연 계획 및 코드 리뷰

### 핵심 메시지
시연은 가입→생성→모임/투표→장부 확인의 정상 플로우와 핵심 예외 코드 재현까지 포함한다.

### 말하기 대본
시연은 "가입/충전 → 챌린지 생성 → 모임/투표 → 장부 확인"의 정상 흐름으로 시작합니다.  
이후 운영 리스크를 보여주기 위해 두 가지 예외를 재현합니다.  
첫째, 모임 비참석자가 지출 투표를 시도하면 `VOTE_007`로 차단되는 흐름입니다.  
둘째, Django 연동 장애 상황에서 그래프 조회가 `LEDGER_004`, HTTP 503으로 반환되는 흐름입니다.  
또한 코드 품질 운영은 변경 단위 리뷰와 페어 리뷰를 결합해, 기능 확장보다 회귀 방지에 우선순위를 두고 진행했습니다.

### 기획↔구현 병기
- 기획 문서의 에러 코드 설명(예: 일부 400/404 표기)과 달리, 실제 구현/운영 기준 시연은 `VOTE_007=403`, `LEDGER_004=503`으로 고정합니다.
- 시연 시나리오는 UX 시나리오 문맥을 참고하되, 실행 절차는 `OUTPUT_09~12`의 구현 기반 흐름으로 맞춥니다.
